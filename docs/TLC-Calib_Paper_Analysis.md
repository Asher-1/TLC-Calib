## 一、研究背景与动机

### 1.1 核心问题

在 LiDAR-Camera 多传感器系统中，传感器之间的**外参 (extrinsics)** 标定至关重要。外参描述了各传感器之间的相对位姿关系（旋转 R + 平移 t），即 camera-to-lidar 的 4×4 齐次变换矩阵 T_c2l。然而现实中面临两大挑战：

1. **Target-based（基于标定板）方法**需要棋盘格、球形反射器等物理标靶，部署成本高，远距离精度差
2. **外参漂移问题**：即便精心标定过，机械振动、热膨胀、物理撞击也会导致外参随时间偏移，需要周期性重新标定

### 1.2 现有无标靶方法的不足

| 方法类别 | 代表工作 | 主要缺陷 |
|----------|----------|----------|
| 边缘对齐 | Levinson et al. [10] | 受限于特征提取质量 |
| 深度学习 | RegNet [13], LCCNet [14] | 需要大规模标注数据，泛化能力差 |
| NeRF-based | INF [15], MOISST [16], UniCal [17] | 隐式体积表示，计算开销极高（>4h） |
| 3DGS-based | 3DGS-Calib [23] | 依赖 hash-grid 编码，对超参数敏感 |
| 2DGS-based | RobustCalib [24] | 依赖表面法线估计，稀疏 LiDAR 下退化 |

**最关键的共性缺陷**：3DGS-Calib [23] 和 RobustCalib [24] 都**只在 LiDAR 可观测区域建模**，将高斯固定在 LiDAR 点上，导致天空、建筑顶部等 LiDAR 盲区没有梯度回传，丢弃了大量有价值的光度监督信号。

### 1.3 论文核心贡献

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                          TLC-Calib 四大核心贡献                                 │
```

## 二、方法总览

TLC-Calib 的核心思想：**基于 Neural Gaussian Splatting 构建全可微的场景表示，联合优化场景表示与传感器外参。** 系统以 LiDAR 作为参考传感器，其坐标系为全局参考系。

```
    ──── Refine (10K, 可选) ──────────────────────────────┤           │
```

## 三、核心技术详解

### 3.1 位姿可微分光栅化 (Pose-Differentiable Rasterization)

**论文公式 (1)-(3)**：通过链式法则将光度损失的梯度传播到相机位姿。

渲染的颜色（标准 alpha compositing）：

```
C(u) = Σ_i c_i · α_i · G^{2D}_i(u) · Π_{j<i} (1 - α_j · G^{2D}_j(u))
```

位姿梯度的链式法则（**论文公式 (2)**）：

```
∂L/∂T_c = Σ_i ( ∂L/∂μ^{2D}_i · ∂μ^{2D}_i/∂μ^c_i · ∂μ^c_i/∂T_c
              + ∂L/∂Σ^{2D}_i · ∂Σ^{2D}_i/∂T_c
              + ∂L/∂c_i · ∂c_i/∂T_c )
```

位姿更新在 SE(3) 李群上进行（**论文公式 (3)**）：

```
T_c ← exp(-η · ∂L/∂τ) · T_c,   τ = [ρ, θ] ∈ se(3)
```

**代码对应**：仓库使用定制的 CUDA 光栅化器 `diff-gaussian-rasterization-w-pose`（后缀 `-w-pose` 区别于标准版），在 `gaussian_renderer/__init__.py` 中传入位姿增量参数 `theta`（旋转）和 `rho`（平移），在 CUDA 层面实现位姿梯度的解析计算。

SE(3) 指数映射的 Python 实现位于 `utils/pose_utils.py::SE3_exp()`，其中 `SO3_exp` 使用 Rodrigues 公式，`SO3_left_jacobian` 计算左 Jacobian 矩阵。

### 3.2 Anchor Gaussians（锚点高斯）

**论文 Section III-C.1**：从聚合的 LiDAR 点云中选取可靠点作为锚点高斯，其**位置在训练过程中保持冻结**，提供全局尺度约束，防止标定漂移。

每个 Anchor 携带以下参数：

| 参数 | 维度 | 是否可学习 | 说明 |
|------|------|-----------|------|
| `_anchor` | [N, 3] | **冻结** (lr=0) | 3D 位置，直接取自 LiDAR 点坐标 |
| `_anchor_feat` | [N, 32] | 可学习 | 局部几何特征向量 |
| `_offset` | [N, K, 3] | 可学习 | K 个辅助高斯的偏移量 |
| `_scaling` | [N, 6] | 可学习 | 各向异性尺度 (3 anchor + 3 offset) |
| `_rotation` | [N, 4] | 不可学习 | 四元数旋转 |
| `_opacity` | [N, 1] | 不可学习 | 不透明度 |

**冻结实现**：在 `arguments/__init__.py` 中，`position_lr_init = 0.0`，`position_lr_final = 0.0`，使 anchor 位置永远不会被梯度更新。

```
关键代码 (arguments/__init__.py):
    self.position_lr_init = 0.0   ← anchor 位置冻结
    self.position_lr_final = 0.0  ← 始终为零

关键代码 (gaussian_model.py::training_setup):
    {'params': [self._anchor], 'lr': training_args.position_lr_init * self.spatial_lr_scale, "name": "anchor"}
    ↑ 0.0 × any_scale = 0.0，梯度不更新
```

**浮点剪除**：低不透明度的锚点被当作 "floaters"，在 `gaussian_model.py::adjust_anchor()` 中根据 `opacity_accum < min_opacity * anchor_demon` 条件剪除。

### 3.3 Auxiliary Gaussians（辅助高斯）

**论文 Section III-C.2**：TLC-Calib 区别于先前方法的核心创新。每个 anchor 通过轻量级 MLP 生成 K 个辅助高斯。

辅助高斯的中心位置：`m_{i,k} = v_i + δ_{i,k} ⊙ s_i`

三个 MLP 的架构（均为两层 + 激活函数）：

```
┌──────────────────────────────────────────────────────────────────────┐
│                      Auxiliary Gaussian MLP 架构                      │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  输入: [anchor_feat(32), view_direction(3)] = 35 维                  │
│                    ↓                                                 │
│  ┌────────────────────────────────────────────┐                      │
│  │ mlp_opacity:                                │                      │
│  │   Linear(35, 32) → ReLU → Linear(32, K)    │                      │
│  │   → Tanh (输出范围 [-1, 1])                  │                      │
│  │   物理含义: 每个辅助高斯的不透明度             │                      │
│  └────────────────────────────────────────────┘                      │
│                                                                      │
│  ┌────────────────────────────────────────────┐                      │
│  │ mlp_cov:                                    │                      │
│  │   Linear(35, 32) → ReLU → Linear(32, 7K)   │                      │
│  │   无激活函数                                  │                      │
│  │   物理含义: scale(3) + quaternion(4) = 7/辅助  │                     │
│  └────────────────────────────────────────────┘                      │
│                                                                      │
│  ┌────────────────────────────────────────────┐                      │
│  │ mlp_color:                                  │                      │
│  │   Linear(35, 32) → ReLU → Linear(32, 3K)   │                      │
│  │   → Sigmoid (输出范围 [0, 1])                │                      │
│  │   物理含义: RGB 颜色                          │                      │
│  └────────────────────────────────────────────┘                      │
│                                                                      │
│  后处理 (gaussian_renderer/__init__.py):                              │
│    scaling = anchor_scale[:,3:] × sigmoid(mlp_output[:,:3])          │
│    rotation = F.normalize(mlp_output[:,3:7])                         │
│    xyz = anchor_pos + offset × anchor_scale[:,:3]                    │
└──────────────────────────────────────────────────────────────────────┘
```

**辅助高斯的四重作用**：
1. **填补 LiDAR 盲区**（天空、建筑顶部），保留全幅图像的光度监督
2. **视角依赖建模**（MLP 输入包含观察方向），提高渲染质量
3. **平滑损失面**，避免位姿陷入局部最优（详见 [13.1 节](#131-损失面平滑化机制)）
4. **opacity mask** 自动过滤无效高斯（`neural_opacity > 0`），动态调整有效点数

### 3.4 Camera Rig 优化策略

**论文 Section III-D**：逐图像顺序更新的 camera rig 优化。

```
┌──────────────────────────────────────────────────────────────────────┐
│                     Camera Rig 优化数据流                              │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  use_rig=True 时:                                                    │
│                                                                      │
│  相机 0 的所有帧 ─┐                                                   │
│                   ├──→ opt_id = 0 ──→ θ₀, ρ₀ (共享6维增量)           │
│  相机 0 的所有帧 ─┘                                                   │
│                                                                      │
│  相机 1 的所有帧 ─┐                                                   │
│                   ├──→ opt_id = 1 ──→ θ₁, ρ₁ (共享6维增量)           │
│  相机 1 的所有帧 ─┘                                                   │
│                                                                      │
│  每次迭代:                                                            │
│  1. 随机采样一帧 (cam_id=c, frame=t)                                  │
│  2. 渲染 I'_{c,t} 时使用 θ_c, ρ_c                                    │
│  3. 计算 L_total, backward()                                          │
│  4. calib_optimizer[c].step() 更新 θ_c, ρ_c                          │
│  5. SE3_exp([ρ_c, θ_c]) 转为 4×4 矩阵                                │
│  6. accum_P[c] = ΔT × accum_P[c] (累积增量)                          │
│  7. θ_c ← 0, ρ_c ← 0 (清零,准备下次迭代)                             │
│                                                                      │
│  关键: 增量清零是为了保持 se(3) 增量在原点附近,                         │
│        小角度近似更准确,梯度更稳定                                      │
└──────────────────────────────────────────────────────────────────────┘
```

**代码实现关键点**（`train.py` + `scene/poses.py`）：

1. **共享外参**：`use_rig=True` 时，`opt_cam_num = 物理相机数`（如 KITTI-360 的 4 个），同一相机的所有帧共享一个 `cam_rot_deltas[opt_id]` 和 `cam_trans_deltas[opt_id]`
2. **增量累积**：每次迭代通过 `SE3_exp` 转换增量为 4×4 矩阵，累积到全局位姿 `accum_P`，然后将增量参数清零
3. **Warmup 机制**：`min_viewpoint_cycle=5`，确保所有训练视角至少被场景优化器看过 5 轮后才释放位姿参数
4. **每个相机独立的优化器**：`calib_optimizer` 是一个列表，每个相机/rig 有独立的 Adam 优化器

### 3.5 自适应体素控制 (Adaptive Voxel Control, AVC)

**论文 Section III-E**：通过二分搜索自动确定体素大小 ε*。

```
ε* = argmin_ε |V(ε) - V_target|
V_target = β × D_traj
D_traj = Σ_{t=1}^{T-1} ||T^L_{t+1} - T^L_t||_2
```

```
┌──────────────────────────────────────────────────────────────────────┐
│                      AVC 二分搜索算法                                 │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  输入: LiDAR 点云 P, 目标体素数 V_target, β=5000                     │
│                                                                      │
│  1. 计算轨迹长度:                                                     │
│     D_traj = Σ ||T^L_{t+1}[:3,3] - T^L_t[:3,3]||                    │
│                                                                      │
│  2. 计算目标: V_target = β × D_traj                                   │
│                                                                      │
│  3. 二分搜索 ε ∈ [0.1, 0.5]:                                         │
│     while |V(ε) - V_target| / V_target > 5%:                         │
│       mid = (lo + hi) / 2                                             │
│       n = count_voxels(P, mid)                                        │
│       if n > V_target: lo = mid  (体素太密,增大ε)                      │
│       else:            hi = mid  (体素太疏,减小ε)                      │
│                                                                      │
│  4. 输出: 最优体素大小 ε*                                              │
│                                                                      │
│  示例:                                                                │
│  KITTI-360 (长直线,D_traj≈200m): ε* ≈ 0.15-0.20                      │
│  FAST-LIVO2 (室内小场景,D_traj≈50m): ε* ≈ 0.10                       │
└──────────────────────────────────────────────────────────────────────┘
```

AVC 的重要性：过小的 ε 导致 anchor 过密（优化不稳定），过大的 ε 降低几何覆盖率（精度退化）。

### 3.6 损失函数

**论文公式 (8)-(9)**：

```
L_total = (1 - λ_dssim) · L_photo + λ_dssim · L_ssim + λ_scale · L_scale
L_photo = L1(I', I)   或   L2(I', I)
L_ssim  = 1 - SSIM(I', I)
L_scale = (1/|V|) · Σ_{i∈V} max(max(s_i)/min(s_i) - σ, 0)
```

默认超参数：
- `lambda_dssim = 0.2`（与标准 3DGS 一致）
- `lambda_scale = 1.0`
- `scale_regularizer (σ) = 10`
- `loss_type = "l1"`

`L_scale` 惩罚过度各向异性的高斯（max/min 比值超过 σ=10），防止退化的细长高斯干扰优化稳定性。这是 TLC-Calib 新增的正则化项，标准 3DGS 和 Scaffold-GS 均没有。

### 3.7 位姿学习率调度

**TLC-Calib 使用余弦退火调度器**（不同于标准 3DGS 的指数衰减）：

```
lr(t) = lr_final + 0.5 × (lr_init - lr_final) × (1 + cos(π × t/T))
```

| 参数 | 初始值 | 终值 | 调度类型 | 衰减比 |
|------|--------|------|---------|--------|
| 旋转 lr | 2×10⁻³ | 2×10⁻⁴ | 余弦退火 | 0.1× |
| 平移 lr | 5×10⁻³ | 5×10⁻⁴ | 余弦退火 | 0.1× |
| 场景各参数 | 各不相同 | 各不相同 | **指数衰减** | - |

场景权重衰减策略：前 15K 迭代 `weight_decay=1e-2`，15K 后关闭（设为 0），确保前半段不过拟合，后半段允许精细调整。

**为什么位姿用余弦退火而场景用指数衰减？** 余弦退火在训练末期衰减更缓慢，位姿参数可以在后期继续做精细微调。指数衰减对场景参数更合适，因为场景需要更快稳定。

### 3.8 From-LiDAR 初始化

初始化协议：相机位姿从 LiDAR 位姿推导，使用基础的 camera-to-lidar 变换矩阵。

```
BASE_CAMERA_TO_LIDAR = [         各数据集额外 yaw 偏移:
    [0,  0,  1,  0],            ┌────────────────────────────────┐
    [-1, 0,  0,  0],            │ KITTI-360: cam2=+90°, cam3=-90°│
    [0,  -1, 0,  0],            │ Waymo:     cam1=+45°, cam2=-45°│
    [0,  0,  0,  1]             │            cam3=+90°, cam4=-90°│
]                                │ FAST-LIVO2: 无偏移             │
                                 └────────────────────────────────┘
相机 z 轴 → LiDAR x 轴 (前方)
相机 x 轴 → LiDAR -y 轴 (左方)
相机 y 轴 → LiDAR -z 轴 (下方)
```

这种初始化产生了论文中提到的"平移误差可达 1.2m"的挑战性场景。

## 四、CUDA 光栅化器位姿梯度传播深度分析

### 4.1 定制光栅化器架构

TLC-Calib 使用的 `diff-gaussian-rasterization-w-pose` 是基于标准 3DGS 光栅化器的改进版本，核心修改是在 CUDA kernel 中增加了对位姿增量 `τ = [ρ, θ]`（6 维 se(3) 向量）的梯度计算。

```
文件结构:
submodules/diff-gaussian-rasterization-w-pose/
├── cuda_rasterizer/
│   ├── forward.cu         # 前向: 3D→2D 投影, alpha compositing
│   ├── backward.cu        # 反向: 梯度计算 (核心修改)
│   ├── math.h             # SE3/SO3 李群运算 (新增)
│   ├── auxiliary.h        # 辅助函数
│   ├── forward.h / backward.h
│   └── rasterizer.h / rasterizer_impl.cu
├── rasterize_points.cu    # PyTorch ↔ CUDA 接口
└── diff_gaussian_rasterization_w_pose/
    └── __init__.py        # Python 绑定, 提取 grad_theta/grad_rho
```

### 4.2 前向传播

前向传播与标准 3DGS 几乎相同，关键步骤：

1. **预处理**（`forward.cu::preprocessCUDA`）：将 3D 高斯投影到 2D 图像平面
   - 使用 `viewmatrix`（世界到相机变换）将高斯中心变换到相机坐标系
   - 计算 2D 协方差矩阵（EWA Splatting）
   - 求逆得到 conic 矩阵
   
2. **渲染**（`forward.cu::renderCUDA`）：Alpha compositing
   - 按深度排序的高斯逐层混合
   - 标准 3DGS 的前后排序渲染

**关键区别**：前向传播时位姿增量 `theta/rho` 不直接参与计算（当增量为零时变换为恒等），而是在反向传播时通过解析 Jacobian 计算梯度。

### 4.3 反向传播 — 梯度的四条路径

位姿梯度 `dL/dτ` 通过**四条**独立路径累积，每个高斯贡献 6 维梯度 `dL_dtau[6*idx + 0..5]`：

```
┌──────────────────────────────────────────────────────────────────────────┐
│              位姿梯度 dL/dτ 的四条传播路径                                │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  路径 ①: renderCUDA (backward)                                           │
│  ┌─────────────────────┐                                                 │
│  │ dL/dPixel            │                                                │
│  │   ↓ alpha blending   │                                                │
│  │ dL/dColor, dL/dAlpha │                                                │
│  │   ↓                  │                                                │
│  │ dL/dMean2D           │──→ 传给路径②                                   │
│  │ dL/dConic2D          │──→ 传给路径③                                   │
│  │ dL/dDepth            │──→ 传给路径④                                   │
│  └─────────────────────┘                                                 │
│                                                                          │
│  路径 ②: preprocessCUDA — 2D 投影均值 → 位姿                              │
│  ┌───────────────────────────────────────────────────┐                    │
│  │ dL/dτ += dL/dMean2D · ∂Mean2D/∂p_C · ∂p_C/∂τ     │                   │
│  │                                                    │                   │
│  │ p_C = T_CW · p_W  (3D点在相机坐标系)               │                   │
│  │ ∂p_C/∂ρ = I₃ₓ₃  (平移直接作用)                     │                   │
│  │ ∂p_C/∂θ = -[p_C]× (反对称矩阵)                     │                   │
│  │                                                    │                   │
│  │ 使用投影矩阵的解析 Jacobian:                        │                   │
│  │ d_proj_dp_C = [α·a,  0,   β·e]                     │                   │
│  │               [0,    α·b, γ·e]                      │                   │
│  │ 其中 α=1/w, β=-x/w², γ=-y/w²                       │                   │
│  └───────────────────────────────────────────────────┘                    │
│                                                                          │
│  路径 ③: computeCov2DCUDA — 2D 协方差 → 位姿                              │
│  ┌───────────────────────────────────────────────────┐                    │
│  │ dL/dτ += dL/dΣ²ᴰ · ∂Σ²ᴰ/∂t_cam · ∂t_cam/∂τ      │                   │
│  │        + dL/dΣ²ᴰ · ∂Σ²ᴰ/∂W · ∂W/∂θ               │                   │
│  │                                                    │                   │
│  │ 平移部分:                                           │                   │
│  │   dp_C_d_rho = I₃ₓ₃                                │                   │
│  │   dp_C_d_theta = -[p_C]×                            │                   │
│  │                                                    │                   │
│  │ 旋转部分 (对旋转矩阵每列微分):                       │                   │
│  │   ∂W/∂θ = [-[r₁]×, -[r₂]×, -[r₃]×]                │                   │
│  │   其中 r₁,r₂,r₃ 是旋转矩阵的三列                    │                   │
│  └───────────────────────────────────────────────────┘                    │
│                                                                          │
│  路径 ④: preprocessCUDA — 深度 → 位姿                                     │
│  ┌───────────────────────────────────────────────────┐                    │
│  │ dL/dτ += dL/dDepth · ∂depth/∂p_C · ∂p_C/∂τ        │                   │
│  │                                                    │                   │
│  │ depth = p_C.z                                       │                   │
│  │ ∂depth/∂p_C = [0, 0, 1]                             │                   │
│  │ ∂p_C/∂ρ 和 ∂p_C/∂θ 同路径②                          │                   │
│  └───────────────────────────────────────────────────┘                    │
│                                                                          │
│  路径 ⑤ (保留但未激活): computeColorFromSH — SH 颜色 → 位姿               │
│  ┌───────────────────────────────────────────────────┐                    │
│  │ 仅当使用球谐函数时激活                                │                   │
│  │ TLC-Calib 使用 MLP 预计算颜色, SH=None                │                   │
│  │ 但代码保留了此路径以保持通用性                         │                   │
│  │                                                    │                   │
│  │ dL/dτ[0:3] += -dL/dmean (平移方向)                   │                   │
│  └───────────────────────────────────────────────────┘                    │
│                                                                          │
│  最终: dL_dtau[6*idx + 0..5] = 路径② + 路径③ + 路径④ (+ 路径⑤ if SH)     │
└──────────────────────────────────────────────────────────────────────────┘
```

### 4.4 梯度聚合与 Python 绑定

所有高斯的 per-point 梯度在 Python 层聚合为全局位姿梯度（`__init__.py`）：

```python
# 反向传播输出 grad_tau: [P, 6] → 求和 → [6]
grad_tau = torch.sum(grad_tau.view(-1, 6), dim=0)
grad_rho = grad_tau[:3].view(1, -1)     # 平移梯度 [1, 3]
grad_theta = grad_tau[3:].view(1, -1)   # 旋转梯度 [1, 3]
```

这实现了论文公式 (2) 的 `Σ_i`，将所有可见高斯的梯度贡献求和。

### 4.5 李代数数学基础（CUDA 实现）

`math.h` 中实现了完整的 SE(3)/SO(3) 李群/李代数运算：

```
┌────────────────────────────────────────────────────────────────────┐
│                    math.h 中的李群/李代数运算                        │
├────────────────────────────────────────────────────────────────────┤
│                                                                    │
│  struct SO3:                                                       │
│    ├── Exp(θ): Rodrigues 公式                                      │
│    │   R = I + sin(||θ||)/||θ|| · [ω]×                             │
│    │     + (1-cos(||θ||))/||θ||² · [ω]×²                          │
│    ├── data(): 返回 mat33 旋转矩阵                                  │
│    └── 乘法运算符: SO3 × SO3                                       │
│                                                                    │
│  struct SE3:                                                       │
│    ├── 从 viewmatrix float[16] 构造                                 │
│    ├── R(): 返回 SO3 旋转                                           │
│    ├── t(): 返回 float3 平移                                        │
│    ├── Exp(τ): T = [R, V·ρ; 0, 1]                                  │
│    │   其中 V 是 SO3 左 Jacobian                                    │
│    └── operator*: SE3 作用于 float3 点                               │
│                                                                    │
│  struct mat33:                                                     │
│    ├── identity(): 单位矩阵                                         │
│    ├── skew_symmetric(v): 反对称矩阵 [v]×                           │
│    ├── transpose(): 转置                                            │
│    └── 矩阵-向量乘法                                                │
│                                                                    │
│  对应 Python 实现 (utils/pose_utils.py):                            │
│    _skew_symmetric(v) → mat33::skew_symmetric                      │
│    _SO3_exp(θ)        → SO3::Exp                                    │
│    _SO3_left_jacobian(θ) → SE3::Exp 内部使用                        │
│    SE3_exp(τ)         → SE3::Exp                                    │
└────────────────────────────────────────────────────────────────────┘
```

## 五、Refine 阶段工作原理

### 5.1 设计动机

标定完成后（30K 迭代），场景表示已经与优化后的外参良好对齐。Refine 阶段的目标是在**冻结位姿**的条件下，进一步提升场景表示的渲染质量（NVS 性能）。

### 5.2 阶段对比

```
┌──────────────────────────────────────────────────────────────────────┐
│                  主训练 vs Refine 阶段对比                             │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  ┌─────────── 主训练阶段 (0-30K) ────────────┐                       │
│  │ • opt_pose = True (位姿可优化)              │                       │
│  │ • Densification: 500-15K (仅剪除)          │                       │
│  │ • Weight decay: 0-15K: 1e-2, 15K后: 0     │                       │
│  │ • 位姿 lr: 余弦退火 (高→低)                 │                       │
│  │ • 保存: iteration 30000                     │                       │
│  └────────────────────────────────────────────┘                       │
│                                                                      │
│  ┌─────────── Refine 阶段 (30K-40K) ──────────┐                       │
│  │ • opt_pose = False (位姿冻结!)              │                       │
│  │ • Densification: 30500-39000               │                       │
│  │   └─ 重新统计梯度, 允许 anchor growing      │                       │
│  │ • Weight decay: 保持 0                     │                       │
│  │ • 位姿 lr: N/A (冻结)                       │                       │
│  │ • 保存: iteration 40000                     │                       │
│  └────────────────────────────────────────────┘                       │
│                                                                      │
│  代码关键:                                                            │
│  if refine_phase:                                                     │
│      opt.start_stat = opt.refine_start        # 30000                │
│      opt.update_from = opt.refine_start + 500 # 30500                │
│      opt.update_until = opt.refine_end - 1000 # 39000                │
│      opt.opt_pose = False                     # 冻结位姿              │
└──────────────────────────────────────────────────────────────────────┘
```

### 5.3 Refine 阶段的 Densification

在 `adjust_anchor` 中，当 `refine=True` 时启用 `anchor_growing`：

1. 计算 offset 梯度累积量的归一化值
2. 在梯度较大的区域（几何细节不足）添加新 anchor
3. 多层级增长：`update_depth=3`，每层阈值递增 `cur_threshold = threshold × (hierachy_factor/2)^i`
4. 随机采样避免过密：`rand_mask > 0.5^(i+1)`
5. 去重：新 anchor 不与已有 anchor 重叠（grid coordinate 检查）

### 6.2 TLC-Calib 独有的创新点

1. **Anchor 冻结策略**：保证 LiDAR 提供的全局尺度不因优化而漂移
2. **Rig 优化**：同一物理相机的所有帧共享一个外参，约束更强
3. **AVC**：根据 LiDAR 轨迹长度自动确定 anchor 密度
4. **尺度正则化 L_scale**：防止各向异性退化的高斯阻碍位姿收敛
5. **Warmup 机制**：先稳定场景表示，再释放位姿优化
6. **AdamW 权重衰减**：场景优化器使用 AdamW 防止过拟合
7. **深度梯度路径**：backward.cu 中新增深度渲染的位姿梯度传播
8. **收敛监控**：实时计算 `||τ||` 范数判断收敛

1. **准备 `params/` 目录**中的标定文件

5. 运行时指定 `--dataset <your_dataset_name>`

```python
viewer.lock.release()         # 释放锁
```

## 十、nvs_eval/ 子模块：标准 3DGS 评估管线

### 10.1 设计理念

`nvs_eval/` 是一个独立的**标准 3DGS** 实现，与 TLC-Calib 的 Neural Gaussian 架构完全不同。这是论文公平对比策略的核心：

```
┌──────────────────────────────────────────────────────────────────────┐
│              nvs_eval/ vs TLC-Calib 主代码对比                        │
├─────────────────┬──────────────────────┬─────────────────────────────┤
│     方面        │    nvs_eval/          │    TLC-Calib 主代码          │
├─────────────────┼──────────────────────┼─────────────────────────────┤
│ Gaussian 类型   │ 标准 3DGS (flat)      │ Anchor + Auxiliary          │
│ 颜色表示        │ 球谐函数 (SH)         │ MLP                         │
│ 位姿优化        │ 无                    │ SE(3) rig 优化              │
│ 光栅化器        │ 标准 diff-gaussian-   │ 定制 -w-pose                │
│                 │ rasterization         │                             │
│ Densification   │ 标准 (split/clone/    │ Anchor growing/pruning      │
│                 │ prune/opacity reset)  │                             │
│ 点云初始化      │ 使用标定后外参        │ 从 LiDAR 点云               │
│ 目的            │ 评估标定质量          │ 联合标定+场景表示           │
│ 输入外参        │ prior_pose_path       │ 从 LiDAR 初始化            │
│ iterations      │ 30000                 │ 30000 + 10000(refine)       │
│ 训练/测试分割   │ llffhold=2            │ llffhold=2                  │
└─────────────────┴──────────────────────┴─────────────────────────────┘
```

### 10.2 nvs_eval/train.py 核心流程

```
nvs_eval/train.py:
  ① GaussianModel(sh_degree=0)        # 标准 3DGS
  ② Scene(dataset, gaussians)          # 加载数据 (使用标定后外参)
  ③ training_setup(opt, fix_xyz=False) # xyz 可学习 (不像 TLC-Calib 冻结)
  ④ for iteration in range(30000):
       viewpoint_cam = random_camera()
       render_pkg = render(cam, gaussians, pipe, bg)
       Ll1 = l1_loss(image, gt)
       loss = (1-λ) * Ll1 + λ * (1-ssim)  # 标准损失 (无 L_scale)
       loss.backward()
       if iteration < densify_until:
           densify_and_prune()          # 标准 3DGS densification
       optimizer.step()
  ⑤ render_sets(test_cameras)          # 渲染测试集
  ⑥ evaluate(PSNR, SSIM, LPIPS)       # 计算指标
```

**关键点**：nvs_eval 不做任何位姿优化，完全依赖输入的外参。因此 NVS 质量直接反映标定精度。

### 13.4 AdamW 权重衰减策略

```
┌──────────────────────────────────────────────────────────────────────┐
│                    AdamW 权重衰减策略                                  │
├──────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  TLC-Calib 使用 AdamW (而非 Adam) 作为场景优化器:                      │

│                                     weight_decay=1e-2)               │
│                                                                      │
│  阶段性权重衰减:                                                      │
│  ┌─────────────────────────────────────────────────┐                  │
│  │ iteration 0 ──────── 15000 ──────── 30000       │                  │
│  │                                                  │                  │
│  │  weight_decay = 1e-2   │   weight_decay = 0     │                  │

│                                                                      │
│  为什么用 AdamW 而非 Adam + L2?                                        │
│  AdamW 解耦权重衰减和自适应学习率,                                     │
```

```
│  ④ 仅对可见 anchor 生成 auxiliary gaussians                           │
│                                                                      │
│  效果:                                                                │
│  • 大规模场景中,单帧可能只有 30-50% 的 anchor 可见                     │
│  • 跳过不可见 anchor 的 MLP 计算,显著减少计算量                        │
│  • visible_filter 使用简化的 3D 协方差投影,比完整渲染快得多             │
│                                                                      │
│  代码: gaussian_renderer/__init__.py::prefilter_voxel()                │
│  调用: voxel_visible_mask = prefilter_voxel(cam, gs, pipe, bg)        │
│        render(..., visible_mask=voxel_visible_mask)                    │
└──────────────────────────────────────────────────────────────────────┘
```

### 13.8 深度渲染与梯度传播

TLC-Calib 在标准 3DGS 基础上新增了**深度图渲染**和相应的位姿梯度路径：

```
前向: depth(u) = Σ_i d_i · α_i · G^{2D}_i(u) · Π_{j<i}(1 - α_j · G^{2D}_j(u))
      其中 d_i = p_C.z (高斯中心在相机坐标系的 z 坐标)

反向 (backward.cu::preprocessCUDA):
  dL_dpCz = dL_ddepth[idx]
  // 深度对3D均值的梯度
  dL_dmeans[idx].x += dL_dpCz * viewmatrix[2]
  dL_dmeans[idx].y += dL_dpCz * viewmatrix[6]
  dL_dmeans[idx].z += dL_dpCz * viewmatrix[10]
  
  // 深度对位姿的梯度 (路径④)
  for i in 0..2:
    dL_dtau[6*idx + i]   += dL_dpCz * dp_C_d_rho.cols[i].z
    dL_dtau[6*idx + i+3] += dL_dpCz * dp_C_d_theta.cols[i].z
```

深度梯度路径虽然在当前损失函数中没有显式使用深度监督，但它通过 alpha compositing 的反向传播间接影响位姿梯度，为位姿优化提供额外的几何约束。

## 十四、CUDA 前向渲染管线深度解析

> **源码路径**: `submodules/diff-gaussian-rasterization-w-pose/cuda_rasterizer/forward.cu`  
> **关联文件**: `rasterizer_impl.cu` (调度层), `auxiliary.h` (辅助函数), `config.h` (常量), `math.h` (Lie 群数学)

### 14.1 Tile-Based 光栅化总体架构

TLC-Calib 的前向渲染管线严格遵循原版 3DGS 的 Tile-Based Rasterization 架构，但在关键位置注入了位姿参数（`theta`, `rho`）以支持可微分标定。整体流程分为三个核心阶段：

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                    CUDA Tile-Based 前向渲染管线                                │
├──────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  Stage 1: preprocessCUDA (逐 Gaussian, 256 threads/block)                    │
│  ┌─────────────────────────────────────────────────────────────────────┐     │
│  │  World Space 3D 点 ──→ Frustum 剔除 ──→ 投影到 NDC                 │     │
│  │  ──→ 计算 3D 协方差 ──→ 投影到 2D 协方差 ──→ 求逆得 Conic         │     │
│  │  ──→ 计算特征值 ──→ Bounding Box (tile 覆盖) ──→ 记录深度         │     │
│  │  ──→ 可选: SH→RGB (TLC-Calib 使用 MLP 预计算, 跳过此步)          │     │
│  └─────────────────────────────────────────────────────────────────────┘     │
│                              ↓                                               │
│  Stage 2: 排序与 Tile 分配 (CPU + GPU 联合)                                  │
│  ┌─────────────────────────────────────────────────────────────────────┐     │
│  │  tiles_touched[] ──→ InclusiveSum (前缀和)                         │     │
│  │  ──→ duplicateWithKeys: 生成 [tile_id | depth] 键值对              │     │
│  │  ──→ RadixSort (按键排序) ──→ identifyTileRanges                  │     │
│  └─────────────────────────────────────────────────────────────────────┘     │
│                              ↓                                               │
│  Stage 3: renderCUDA (逐 Tile, 16×16=256 threads/block)                     │
│  ┌─────────────────────────────────────────────────────────────────────┐     │
│  │  每个 tile 的线程块协作加载 Gaussians ──→ 逐像素 alpha-blending    │     │
│  │  ──→ 颜色 + 深度 + 不透明度 ──→ 背景混合 ──→ 写出帧缓冲         │     │
│  └─────────────────────────────────────────────────────────────────────┘     │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

**配置常量** (`config.h`):
- `BLOCK_X = 16`, `BLOCK_Y = 16` → 每 tile 覆盖 16×16 = 256 像素
- `NUM_CHANNELS = 3` → RGB 三通道
- `BLOCK_SIZE = 256` → 每个线程块处理一个 tile

### 14.2 preprocessCUDA: 逐 Gaussian 预处理内核

这是前向管线中计算最密集的阶段，每个线程处理一个 Gaussian。

**Step 1: Frustum 剔除**
```c
// auxiliary.h: in_frustum()
float3 p_view = transformPoint4x3(p_orig, viewmatrix);
if (p_view.z <= 0.2f)  // 近平面剔除阈值
    return false;
```
- 将 3D 点变换到相机坐标系，检查是否在近平面 (z > 0.2) 之后
- 未通过的点标记 `radii[idx] = 0`，后续全部跳过

**Step 2: 透视投影到 NDC**
```c
float3 p_orig = {orig_points[3*idx], orig_points[3*idx+1], orig_points[3*idx+2]};
float4 p_hom = transformPoint4x4(p_orig, projmatrix);  // 齐次坐标
float p_w = 1.0f / (p_hom.w + 0.0000001f);            // 避免除零
float3 p_proj = {p_hom.x * p_w, p_hom.y * p_w, p_hom.z * p_w};  // NDC
```
- `projmatrix` = `viewmatrix * projection_matrix`，由 Python 端预乘传入
- 注意：此处的 `projmatrix` 已经包含了当前帧的位姿增量 delta（在 Python 端通过 `update_pred_pose` 应用）

**Step 3: 3D 协方差矩阵计算** (`computeCov3D`)
```c
// 从 scale + quaternion → 3D 协方差矩阵
glm::mat3 S = diag(mod * scale.x, mod * scale.y, mod * scale.z);
glm::mat3 R = quaternion_to_matrix(rot);  // 四元数→旋转矩阵
glm::mat3 M = S * R;
glm::mat3 Sigma = transpose(M) * M;      // 正半定保证
// 仅存储上三角 (6个元素): Sigma[0:5]
```

**Step 4: 3D→2D 协方差投影** (`computeCov2D`)
```
数学推导 (EWA Splatting, Zwicker et al., 2002):

已知:
  - 3D 协方差 Σ_3D (世界坐标系)
  - 视图矩阵 W (world → camera)
  - 投影雅可比 J (camera → image)

投影:
  T = W × J
  Σ_2D = T^T × Σ_3D × T

实现:
  (1) 变换到相机坐标: t = W × p_world
  (2) 计算 Jacobian:
      J = | fx/tz    0    -fx*tx/tz²  |
          |   0    fy/tz  -fy*ty/tz²  |
          |   0      0        0       |
  (3) 组合: T = W × J
  (4) 投影: cov2D = T^T × Vrk × T
  (5) 低通滤波: cov2D[0][0] += 0.3; cov2D[1][1] += 0.3
```
- **视野限制** (`limx = 1.3 * tan_fovx`): 对视野外的点进行 clamp，防止极端投影
- **低通滤波** (+0.3): 保证每个 Gaussian 至少覆盖约 1 个像素，避免子像素锯齿

**Step 5: Conic (逆协方差) 计算**
```c
float det = cov.x * cov.z - cov.y * cov.y;  // 行列式
float3 conic = {cov.z/det, -cov.y/det, cov.x/det};  // 2×2 对称矩阵求逆
```
- 行列式为零时整个 Gaussian 被跳过（退化椭圆）

**Step 6: 屏幕覆盖范围计算**
```c
// 特征值 → 3σ 半径
float mid = 0.5f * (cov.x + cov.z);
float lambda1 = mid + sqrt(max(0.1f, mid*mid - det));
float lambda2 = mid - sqrt(max(0.1f, mid*mid - det));
float my_radius = ceil(3.f * sqrt(max(lambda1, lambda2)));

// NDC → 像素坐标
float2 point_image = {ndc2Pix(p_proj.x, W), ndc2Pix(p_proj.y, H)};
// ndc2Pix: ((v + 1.0) * S - 1.0) * 0.5

// 计算覆盖的 tile 矩形范围
getRect(point_image, my_radius, rect_min, rect_max, grid);
tiles_touched[idx] = (rect_max.y - rect_min.y) * (rect_max.x - rect_min.x);
```

**Step 7: 颜色计算分支**
```c
if (colors_precomp == nullptr) {
    // 标准 3DGS: SH → RGB（最多 4 阶, 16 系数/通道）
    glm::vec3 result = computeColorFromSH(idx, D, M, means, campos, shs, clamped);
} else {
    // TLC-Calib: 使用 MLP 预计算的颜色 (Python 端 generate_neural_gaussians)
    // colors_precomp 已包含 RGB，直接跳过 SH
}
```

**关键输出**:
| 输出 | 大小/Gaussian | 用途 |
|------|---------------|------|
| `depths[idx]` | float | 前后排序 |
| `radii[idx]` | int | 可见性判断 |
| `points_xy_image[idx]` | float2 | 像素坐标 |
| `conic_opacity[idx]` | float4 | Conic 3个 + opacity 1个打包 |
| `tiles_touched[idx]` | uint32 | 覆盖 tile 数 |

### 14.3 renderCUDA: 逐像素 Alpha-Blending 内核

这是最终的像素着色阶段，每个线程处理一个像素。

```
线程块组织:
  - grid:   (ceil(W/16), ceil(H/16))  → 每个 block 对应一个 tile
  - block:  (16, 16)                  → 每个 thread 对应一个像素

共享内存 (per tile):
  - collected_id[256]          → Gaussian 全局 ID
  - collected_xy[256]          → 2D 投影坐标
  - collected_conic_opacity[256] → Conic + opacity
  - collected_depth[256]       → 深度值
```

**Alpha-Blending 核心循环**:
```c
float T = 1.0f;        // 累积透射率，初始为 1（完全透明）
float C[3] = {0};      // 累积颜色
float D = 0.0f;        // 累积深度

for (每一批 Gaussians in sorted order) {
    // 协作加载: 全部 256 线程一起从全局内存加载到共享内存
    if (range.x + progress < range.y) {
        collected_id[tid] = point_list[range.x + progress];
        collected_xy[tid] = points_xy_image[coll_id];
        // ...
    }
    __syncthreads();  // 同步等待加载完成

    for (每个 Gaussian j in batch) {
        // 计算像素到 Gaussian 中心的 Mahalanobis 距离
        float2 d = {xy.x - pixf.x, xy.y - pixf.y};
        float power = -0.5f * (con_o.x*d.x*d.x + con_o.z*d.y*d.y) - con_o.y*d.x*d.y;
        if (power > 0.0f) continue;  // 在 Gaussian 外部

        // Alpha = opacity × Gaussian 衰减
        float alpha = min(0.99f, con_o.w * exp(power));
        if (alpha < 1.0f/255.0f) continue;  // 贡献太小

        // 提前终止检查
        float test_T = T * (1 - alpha);
        if (test_T < 0.0001f) { done = true; continue; }

        // Alpha-compositing (前到后)
        for (int ch = 0; ch < 3; ch++)
            C[ch] += features[id * 3 + ch] * alpha * T;
        D += depth[j] * alpha * T;  // 深度也做 alpha 混合

        // 统计被多少像素覆盖 (用于 densification)
        if (test_T > 0.5f)
            atomicAdd(&n_touched[id], 1);

        T = test_T;
    }
}

// 最终: 加入背景色
for (int ch = 0; ch < 3; ch++)
    out_color[ch * H * W + pix_id] = C[ch] + T * bg_color[ch];
out_depth[pix_id] = D;
out_opacity[pix_id] = 1 - T;
```

**关键设计细节**:

1. **前到后排序渲染** (Front-to-Back): 排序键为 `[tile_id | depth]`，保证同一 tile 内按深度从近到远遍历
2. **早停机制**: 当 `T < 0.0001` 时停止混合，减少无效计算
3. **整 block 投票**: `__syncthreads_count(done)` — 全部线程都完成时才跳过整个 batch
4. **n_touched 统计**: `T > 0.5` 时才计数，用于后续 anchor growing/pruning 的 densification 判据
5. **深度渲染**: 与颜色完全并行地做 alpha-blending，为反向传播提供深度梯度路径

### 14.4 排序与分 Tile 调度

排序阶段由 `rasterizer_impl.cu` 中的 `Rasterizer::forward()` 编排：

```
Step 1: InclusiveSum (前缀和)
  tiles_touched = [2, 3, 0, 2, 1]
  point_offsets = [2, 5, 5, 7, 8]  ← 每个 Gaussian 写入的起始偏移

Step 2: duplicateWithKeys
  对每个 Gaussian 的每个覆盖 tile，生成:
    key = (tile_y * grid.x + tile_x) << 32 | depth_as_uint32
    value = gaussian_idx

Step 3: RadixSort (CUB 库)
  按 key 排序 → 同一 tile 内按深度排列

Step 4: identifyTileRanges
  扫描排序后的 keys，标记每个 tile 的 [start, end) 范围
  ranges[tile_id] = {start_idx, end_idx}
```

**内存布局 — 三类状态缓冲区**:
| 缓冲区 | 生命周期 | 内容 |
|---------|----------|------|
| `GeometryState` | 逐 Gaussian | depths, means2D, cov3D, conic_opacity, rgb, tiles_touched |
| `BinningState` | 排序中间态 | key/value 对(未排序+已排序), sorting workspace |
| `ImageState` | 逐像素 | accum_alpha (final_T), n_contrib, tile ranges |

### 14.5 filter_preprocessCUDA: 体素预过滤专用内核

这是 TLC-Calib 用于 `prefilter_voxel()` 的轻量级预处理内核：

```c
// 与 preprocessCUDA 相比:
// ✓ Frustum 剔除
// ✓ 3D→2D 协方差投影
// ✓ 特征值 → 半径计算
// ✗ 不计算颜色 (无 SH/color_precomp)
// ✗ 不计算深度
// ✗ 不计算 tiles_touched
// ✗ 不存储 conic_opacity
```

唯一输出是 `radii[idx]`，非零即可见。Python 端据此生成 `visible_mask`：
```python
# gaussian_renderer/__init__.py
radii_pure = rasterizer.visible_filter(means3D=means3D, scales=scales[:,:3], ...)
return radii_pure > 0  # → bool tensor, 传给 generate_neural_gaussians
```

这使得 Neural Gaussian 生成只在可见 anchor 上运行，节省 30-50% 计算量。

### 14.6 前向与反向的对称设计

```
前向 (forward.cu)                          反向 (backward.cu)
┌──────────────────────────┐               ┌──────────────────────────┐
│ preprocessCUDA           │               │ BACKWARD::render         │
│   computeCov3D           │  ←──对称──→   │   renderCUDA (反向)      │
│   computeCov2D           │               │ BACKWARD::preprocess     │
│   computeColorFromSH     │               │   computeCov2DCUDA       │
│ renderCUDA               │               │   preprocessCUDA (反向)  │
│   alpha-blending         │               │     computeColorFromSH   │
│   depth-blending         │               │     computeCov3D         │
└──────────────────────────┘               └──────────────────────────┘

关键差异:
  反向 renderCUDA: 从后向前遍历 (reverse order)
  反向 preprocessCUDA: 额外计算 dL/dtau (6-DOF 位姿梯度)
  反向 computeCov2DCUDA: 额外计算 dL/d(W) → dL/d(theta) 梯度
```

每个前向计算步骤在反向中都有对应的解析梯度实现——没有依赖 PyTorch 的 autograd，全部在 CUDA 中手写。这是 TLC-Calib 能高效训练的核心原因之一。

### 15.2 render_set 函数详解

核心渲染函数 `render_set()` 的关键步骤：

```python
def render_set(model_path, name, iteration, views, gaussians, pipeline, background):
    # 1. 创建四个输出目录
    compare_path = .../compares
    render_path  = .../renders
    error_path   = .../errors
    gts_path     = .../gt

    for idx, view in enumerate(views):
        # 2. 应用标定后位姿 (关键！)
        update_pred_pose(view, gaussians)

        # 3. 渲染管线 (与 train.py 完全一致)
        voxel_visible_mask = prefilter_voxel(view, gaussians, pipeline, background)
        render_pkg = render(view, gaussians, pipeline, background,
                          visible_mask=voxel_visible_mask)

        # 4. 提取渲染结果
        rendering = torch.clamp(render_pkg["render"], 0.0, 1.0)
        visible_count = (render_pkg["radii"] > 0).sum()

        # 5. 计算绝对误差图
        gt = view.original_image[0:3, :, :]
        errormap = (rendering - gt).abs()

        # 6. 保存四种可视化输出
        save_image(rendering, render_path/...)
        save_image(errormap,  error_path/...)
        save_image(gt,        gts_path/...)
        save_image(cat([rendering, gt], dim=-1), compare_path/...)  # 左右拼接
```

**特别注意**: 
- `update_pred_pose(view, gaussians)` 在渲染前被调用 — 这确保了渲染使用的是**标定后的位姿**而非初始位姿
- `gaussians.eval()` 禁用 dropout/batchnorm，但 TLC-Calib 的 MLP 未使用这些层，主要作用是确保推理一致性
- 计时跳过前 5 帧 (`t_list[5:]`) 以排除 GPU 预热开销

### 16.2 逐方法深度对比

#### A. 3DGS-Calib (Hérau et al., ICRA 2024)

**核心思路**: 将 LiDAR 点固定为 3DGS 中心，使用 hash-grid 编码的 MLP 生成 Gaussian 属性，联合优化位姿和场景。

**与 TLC-Calib 关键差异**:

| 维度 | 3DGS-Calib | TLC-Calib |
|------|-----------|-----------|
| 点位置 | 固定在 LiDAR 点上 | Anchor 固定 + Auxiliary 可学习 |
| 属性生成 | Hash-Grid MLP | 视角自适应 MLP |
| 覆盖范围 | 仅 LiDAR 覆盖区域 | 扩展到 LiDAR 盲区 |
| 密度控制 | 无 | AVC (自适应体素控制) |
| 时空标定 | ✓ (支持时间偏移) | ✓ (支持时间偏移) |
| 超参敏感性 | 高 (hash-grid 编码) | 低 (anchor+MLP 架构) |

**TLC-Calib 论文对此的分析**: 3DGS-Calib 将所有高斯绑定在 LiDAR 点上，导致：
1. **LiDAR 盲区无法渲染** → 天空、远处建筑等区域缺失监督
2. **Hash-grid 编码对场景复杂度敏感** → 不同场景需要调参
3. **无辅助高斯** → 在噪声初始化下容易过拟合错误位姿

#### B. RobustCalib (Zhou et al., RA-L 2025)

**核心思路**: 两阶段策略——先用 2D Gaussian Splatting (2DGS) 从 LiDAR 点云学习几何，再通过重投影和三角化损失标定外参。

**与 TLC-Calib 关键差异**:

| 维度 | RobustCalib | TLC-Calib |
|------|------------|-----------|
| GS 类型 | 2DGS (平面) | 3DGS (体积) |
| 优化策略 | 两阶段解耦 | 联合优化 + Refine |
| 损失函数 | 光度 + 重投影 + 三角化 | 光度 (L1+D-SSIM) + 尺度正则 |
| 法线依赖 | ✓ (2DGS 需要法线) | ✗ |
| Rig 优化 | ✗ (逐帧) | ✓ (共享外参) |
| 稀疏 LiDAR | 退化 (法线估计差) | 鲁棒 (不依赖法线) |

**TLC-Calib 的优势**: RobustCalib 依赖 2DGS 的表面法线质量，在稀疏 LiDAR (如固态 LiDAR) 下性能下降；TLC-Calib 的 Neural Gaussian 架构不依赖法线估计，在 Fast-LIVO2 (固态 LiDAR) 数据上也表现优异。

#### C. HiGS-Calib (Zhang et al., TCSVT 2025)

**核心思路**: 提出 LCPG (Local-Consistent Photometric-Geometric) 误差模型，利用局部窗口内的空间一致性量化位姿偏差，采用分层粗到细的迭代优化策略。

**与 TLC-Calib 关键差异**:

| 维度 | HiGS-Calib | TLC-Calib |
|------|-----------|-----------|
| 优化范式 | 解耦 (先建模后标定) | 联合 (场景+位姿同时优化) |
| 误差模型 | LCPG (几何属性) | 纯光度 + 正则化 |
| 颜色依赖 | 绕开颜色-位姿耦合 | 直接利用颜色梯度 |
| 层级结构 | 粗→细 迭代 | 单阶段 + Refine |
| 外部依赖 | ROMA 光流模型 | 无 (端到端) |
| LiDAR 误差容忍 | ✓ (局部一致性) | ✓ (Anchor 冻结保稳定) |

**HiGS-Calib 的独特贡献**: 解决了联合优化中的"鸡与蛋"困境（精确标定需要精确场景，精确场景需要精确标定），通过解耦策略避免震荡。

**TLC-Calib 的回应**: 通过 Anchor 冻结 + Warmup + Rig 共享 + Refine 阶段，在联合优化框架内解决了同样的问题，且无需外部光流模型依赖。

#### D. CalibAnything (Luo et al., 2024)

**核心思路**: 零训练方法，使用基础模型 (SAM, DINOv2) 提取 LiDAR 和图像的语义特征，通过特征匹配实现标定。

**与 TLC-Calib 关键差异**:

| 维度 | CalibAnything | TLC-Calib |
|------|-------------|-----------|
| 训练需求 | 零训练 | 需要优化 (~11min) |
| 表示方法 | 特征匹配 | Neural Gaussian |
| 精度 | 中等 | 高 |
| 泛化性 | 跨域好 | 同域最优 |
| 速度 | >5h (特征提取) | ~11min |
| 多相机 | 逐相机 | Rig 统一优化 |

#### E. INF (NeRF-based, 2024)

代表了 NeRF 时代的最后改进，速度 >4h，精度被 3DGS 方法全面超越。

### 16.5 TLC-Calib 的差异化优势

**1. 唯一的端到端联合优化 + Rig 约束**

TLC-Calib 是唯一在 3DGS 框架下实现 Rig 优化的方法。所有相机共享外参，每次更新影响全部训练帧，提供了强正则化效果。消融实验显示去掉 Rig 后旋转误差恶化 15×。

**2. 最完整的 CUDA 级位姿梯度**

**3. LiDAR 盲区扩展能力**

**4. 最广泛的传感器覆盖**

唯一在三种不同数据集 (KITTI-360 旋转LiDAR + Waymo 旋转LiDAR + Fast-LIVO2 固态LiDAR) 上验证的方法，证明了跨传感器模态的泛化性。

**5. 完整的评估生态**

提供了 NVS 评估管线 (`metrics_nvs.py`)、位姿评估 (`metrics_pose.py`)、批量评估 (`full_eval.py`) 和实时 Web Viewer，是评估最完整的开源实现。

## 十七、CUDA 反向渲染内核深度解析

> **源码路径**: `submodules/diff-gaussian-rasterization-w-pose/cuda_rasterizer/backward.cu`  
> **本章重点**: 反向 `renderCUDA` 的逆向遍历、梯度分解、位姿梯度的四路汇聚

### 17.1 反向 renderCUDA 的逆向遍历机制

前向渲染从前到后 (front-to-back) 混合，反向渲染则从**后到前 (back-to-front)** 遍历。这是因为 alpha-compositing 的链式法则要求逆序访问。

```
前向渲染 (front-to-back):           反向渲染 (back-to-front):
  range.x ──→ range.y                 range.y ──→ range.x
  G₁ → G₂ → G₃ → ... → Gₙ           Gₙ → ... → G₃ → G₂ → G₁

前向:                                反向:
  T = 1.0                            T = T_final (从前向保存的值)
  for i = 0 → N:                     for i = N → 0:
    C += color[i] * α[i] * T           ∂L/∂α[i], ∂L/∂color[i]
    T *= (1 - α[i])                     T /= (1 - α[i])  ← 关键: 除法恢复
```

**核心数据恢复机制**:

```c
// 反向中从 T_final 恢复每一步的 T
T = T_final;  // 从前向保存

// 在遍历中逆向恢复
T = skip ? T : T / (1.f - alpha);  // 除法恢复前一步的 T
```

**共享内存结构** (与前向对称，但增加了梯度累积缓冲):

```c
// 从前向复用的数据
__shared__ int collected_id[BLOCK_SIZE];
__shared__ float2 collected_xy[BLOCK_SIZE];
__shared__ float4 collected_conic_opacity[BLOCK_SIZE];
__shared__ float collected_colors[C * BLOCK_SIZE];
__shared__ float collected_depths[BLOCK_SIZE];

// 反向新增: 梯度累积共享内存 (用于 warp-level reduce)
__shared__ float2 dL_dmean2D_shared[BLOCK_SIZE];
__shared__ float3 dL_dcolors_shared[BLOCK_SIZE];
__shared__ float dL_ddepths_shared[BLOCK_SIZE];
__shared__ float dL_dopacity_shared[BLOCK_SIZE];
__shared__ float4 dL_dconic2D_shared[BLOCK_SIZE];
```

### 17.2 逐像素梯度分解

对于每个像素-Gaussian 对，反向传播计算以下梯度：

**1. 颜色梯度** `dL/d(color[i])`:
```c
// dchannel_dcolor = α[i] × T
const float dchannel_dcolor = alpha * T;
for (int ch = 0; ch < C; ch++) {
    local_dL_dcolors[ch] = skip ? 0.0f : dchannel_dcolor * dL_dpixel[ch];
}
```

**2. Alpha 梯度** `dL/d(α[i])`:
```c
// 颜色部分: ∂L/∂α = Σ_ch (c[i] - R[i]) × ∂L/∂pixel[ch] × T
float dL_dalpha = 0.0f;
for (int ch = 0; ch < C; ch++) {
    accum_rec[ch] = last_alpha * last_color[ch] + (1 - last_alpha) * accum_rec[ch];
    dL_dalpha += (c - accum_rec[ch]) * dL_dchannel;
}

// 深度部分: 相同的 alpha-compositing 链式法则
dL_dalpha += (depth - accum_rec_depth) * dL_dpixel_depth;

// 乘以 T
dL_dalpha *= T;

// 背景色修正: 考虑 alpha 对背景贡献的影响
float bg_dot_dpixel = Σ(bg_color[i] * dL_dpixel[i]);
dL_dalpha += (-T_final / (1 - alpha)) * bg_dot_dpixel;
```

**3. 2D 均值梯度** `dL/d(mean2D)`:
```c
const float dL_dG = con_o.w * dL_dalpha;  // ∂L/∂G = opacity × ∂L/∂α
const float gdx = G * d.x;
const float gdy = G * d.y;
const float dG_ddelx = -gdx * con_o.x - gdy * con_o.y;  // ∂G/∂Δx
const float dG_ddely = -gdy * con_o.z - gdx * con_o.y;  // ∂G/∂Δy

dL_dmean2D.x = dL_dG * dG_ddelx * (0.5f * W);  // 乘以 NDC→pixel 雅可比
dL_dmean2D.y = dL_dG * dG_ddely * (0.5f * H);
```

**4. Conic 梯度** `dL/d(conic)`:
```c
dL_dconic2D.x = -0.5f * gdx * d.x * dL_dG;  // ∂L/∂conic_xx
dL_dconic2D.y = -0.5f * gdx * d.y * dL_dG;  // ∂L/∂conic_xy
dL_dconic2D.w = -0.5f * gdy * d.y * dL_dG;  // ∂L/∂conic_yy
```

### 17.3 Warp-Level Reduce Sum 优化

TLC-Calib 的反向渲染使用了一个**关键优化** — `render_cuda_reduce_sum`：

```c
template <typename group_t, typename... Lists>
__device__ void render_cuda_reduce_sum(group_t g, Lists... lists) {
    int lane = g.thread_rank();
    g.sync();
    for (int i = g.size() / 2; i > 0; i /= 2) {
        (..., reduce_helper(lane, i, lists));  // C++17 fold expression
        g.sync();
    }
}
```

```
256 线程处理一个 tile 中同一个 Gaussian 的梯度:

Thread[0..255] 各自计算 dL/d(mean2D), dL/d(conic), dL/d(opacity), dL/d(color)

Reduce Sum (log₂(256) = 8 步):
  Step 1: 线程 [0..127] += [128..255]
  Step 2: 线程 [0..63]  += [64..127]
  ...
  Step 8: 线程 [0]      += [1]

Thread[0] 执行 atomicAdd 写入全局内存

优势: 将 256 次 atomicAdd 减少为 1 次 → 减少原子竞争 8×
```

**Skip Counter 优化**: 当整个 block 的所有 256 线程都跳过某个 Gaussian 时，直接 `continue` 避免无效的 reduce 操作。

```c
if (skip) atomicAdd(&skip_counter, 1);
block.sync();
if (skip_counter == BLOCK_SIZE) continue;  // 全部跳过
```

### 17.4 computeCov2DCUDA 的位姿梯度注入

这是位姿梯度的**第二条路径** (2D 协方差 → 位姿)。梯度从 `dL/d(conic)` 出发，经过逆协方差、2D 协方差、投影雅可比 T = W×J，最终到达位姿。

**位姿梯度通过 Jacobian 传播到 t (相机坐标系):**

```c
SE3 T_CW(view_matrix);
mat33 R = T_CW.R().data();
float3 t = transformPoint4x3(mean, view_matrix);  // 重新计算相机坐标

// ∂(p_cam)/∂(ρ) = I (平移对平移的雅可比)
mat33 dpC_drho = mat33::identity();
// ∂(p_cam)/∂(θ) = -[p_cam]× (旋转的 Lie 代数雅可比)
mat33 dpC_dtheta = -mat33::skew_symmetric(t);

// 链式法则: dL/dτ = dL/dt × dt/dτ
float dL_dt[6];
for (int i = 0; i < 3; i++) {
    float3 c_rho = dpC_drho.cols[i];
    float3 c_theta = dpC_dtheta.cols[i];
    dL_dt[i]     = dL_dtx * c_rho.x   + dL_dty * c_rho.y   + dL_dtz * c_rho.z;
    dL_dt[i + 3] = dL_dtx * c_theta.x + dL_dty * c_theta.y + dL_dtz * c_theta.z;
}
// 累加到位姿梯度
for (int i = 0; i < 6; i++)
    dL_dtau[6 * idx + i] += dL_dt[i];
```

**旋转矩阵 W 的梯度 → θ 梯度:**

```c
// W = R (视图矩阵的旋转部分)
// dW/dθ = -[R_col]× (反对称矩阵)

mat33 n_W1_x = -mat33::skew_symmetric(c1);  // c1 = R 的第1列
mat33 n_W2_x = -mat33::skew_symmetric(c2);
mat33 n_W3_x = -mat33::skew_symmetric(c3);

float3 dL_dtheta = {};
dL_dtheta.x = dot(dL_dWc1, n_W1_x.cols[0]) + dot(dL_dWc2, n_W2_x.cols[0])
            + dot(dL_dWc3, n_W3_x.cols[0]);
// ... y, z 类似

dL_dtau[6*idx + 3] += dL_dtheta.x;
dL_dtau[6*idx + 4] += dL_dtheta.y;
dL_dtau[6*idx + 5] += dL_dtheta.z;
```

### 17.5 preprocessCUDA 反向: 位姿梯度的四路汇聚

反向 `preprocessCUDA` 是四条位姿梯度路径的汇聚点。`dL_dtau` 通过 `+=` 从三个独立来源累积：

```
dL_dtau[6*idx + 0..5] 的累积来源:

┌─────────────────────────────────────────────────────────────────────┐
│ 路径 1: 2D 投影均值 (preprocessCUDA 内部)                          │
│   dL/dmean2D → dL/d(proj) → dL/d(p_cam) → dL/dτ                 │
│   数学: dmean2D_dtau[6] = ∂proj/∂p_cam × ∂p_cam/∂τ              │
│   代码: dL_dt[i] = dL_dmean2D.x * dmean2D_dtau[i].x              │
│                   + dL_dmean2D.y * dmean2D_dtau[i].y              │
│   注入: dL_dtau[6*idx + i] += dL_dt[i]                            │
├─────────────────────────────────────────────────────────────────────┤
│ 路径 2: 深度 (preprocessCUDA 内部)                                 │
│   dL/ddepth → dL/d(p_cam.z) → dL/dτ                              │
│   数学: depth = p_cam.z, ∂depth/∂ρ = [0,0,1]^T                   │
│                            ∂depth/∂θ = -[p_cam]×.row(2)          │
│   代码: dL_dtau[6*idx+i]   += dL_dpCz * c_rho.z                  │
│         dL_dtau[6*idx+i+3] += dL_dpCz * c_theta.z                │
├─────────────────────────────────────────────────────────────────────┤
│ 路径 3: 2D 协方差 (computeCov2DCUDA, 独立 kernel)                  │
│   dL/dconic → dL/dcov2D → dL/dT → dL/dJ → dL/d(t) → dL/dτ      │
│                                   → dL/dW → dL/d(θ)              │
│   已在 17.4 节详述                                                 │
├─────────────────────────────────────────────────────────────────────┤
│ 路径 4: 视角相关颜色 (computeColorFromSH, 条件激活)                │
│   dL/dcolor → dL/d(dir) → dL/d(mean) → dL/dτ                    │
│   TLC-Calib 使用 MLP 颜色, 此路径仅在 SH 模式下激活              │
│   代码: dL_dtau[6*idx+0] += -dL_dmean.x                          │
│         dL_dtau[6*idx+1] += -dL_dmean.y                          │
│         dL_dtau[6*idx+2] += -dL_dmean.z                          │
└─────────────────────────────────────────────────────────────────────┘

最终: dL_dtau 是一个 [P, 6] 张量 (P = Gaussian 数, 6 = 位姿自由度)
      在 Python 端聚合为 [1, 6] 后分成 grad_theta[1,3] 和 grad_rho[1,3]
```

### 18.2 位姿梯度的聚合与拆分

这是整个位姿可微分标定的**关键衔接点**：

```python
# CUDA 返回: grad_tau 形状 [P*6] (每个 Gaussian 6个位姿梯度分量)
# P = 当前帧可见 Gaussian 数

# Step 1: reshape 为 [P, 6]
grad_tau = grad_tau.view(-1, 6)

# Step 2: 对所有 Gaussian 求和 → [6]
grad_tau = torch.sum(grad_tau, dim=0)
# 物理含义: 所有 Gaussian 对位姿梯度的贡献叠加
# 这正确反映了: 位姿变化影响所有 Gaussian 的投影

# Step 3: 拆分为旋转和平移梯度
grad_rho   = grad_tau[:3].view(1, -1)  # [1, 3] 平移增量 Lie 代数
grad_theta = grad_tau[3:].view(1, -1)  # [1, 3] 旋转增量 Lie 代数
```

**为什么 CUDA 端是逐 Gaussian 的?**

```
每个 Gaussian 受位姿影响的方式不同:
  - 近处 Gaussian: 位姿变化 → 大的 2D 位移 → 大梯度
  - 远处 Gaussian: 位姿变化 → 小的 2D 位移 → 小梯度
  - 边缘 Gaussian: 被 clamp → 梯度为 0

所以在 CUDA 中逐 Gaussian 计算, 在 Python 中求和是正确的
梯度方向由 Gaussian 分布的加权投票决定
```

## 十九、TLC-Calib 与 NeRF-Based 标定方法对比

### 19.1 NeRF-Based 标定方法概述

在 3DGS 出现之前 (2021-2023)，基于 NeRF 的标定是神经渲染标定的主要范式。代表方法有：

```
NeRF 标定方法时间线:

2021  iNeRF        首次用 NeRF 做位姿估计 (camera-only)
2023  INF          LiDAR-Camera 融合 NeRF, 分离几何/颜色学习
2023  MOISST       多传感器时空标定, 单一 NeRF 场景
2024  SOAC         改进 MOISST, 传感器特定 NeRF + 重叠感知
```

### 19.2 逐方法对比: INF / MOISST / SOAC

#### A. INF (Implicit Neural Fusion, IROS 2023)

**核心思路**: 先用 LiDAR 扫描优化 NeRF 的密度场 (几何)，再通过 360° 相机图像学习颜色字段，同时优化 LiDAR→Camera 外参。

| 维度 | INF | TLC-Calib |
|------|-----|-----------|
| 场景表示 | NeRF (MLP 隐式) | Neural Gaussian (显式点) |
| 渲染方式 | 射线行进 (ray marching) | Tile-based 光栅化 |
| 几何/颜色分离 | ✓ (先几何后颜色) | ✗ (联合优化) |
| 训练时间 | >4h | ~11min |
| 实时渲染 | ✗ | ✓ (>100 FPS) |
| 360° 相机 | ✓ | 针筒相机模型 |
| 多相机 Rig | ✗ | ✓ |

**INF 的局限**: 射线行进的计算成本 O(R×S) (R=射线数, S=采样点数) 远高于光栅化 O(N×T) (N=Gaussian 数, T=tile 覆盖数)。

#### B. MOISST (Multimodal Optimization of Implicit Scene for SpatioTemporal, 2023)

**核心思路**: 为整个传感器系统构建单一 NeRF 场景，利用参考传感器 (通常是 LiDAR) 的轨迹作为基准，联合优化所有传感器的外参和时间偏移。

| 维度 | MOISST | TLC-Calib |
|------|--------|-----------|
| 场景表示 | 单一 NeRF | Neural Gaussian |
| 多传感器 | ✓ (任意 LiDAR/Camera 组合) | ✓ (多 Camera, 单 LiDAR) |
| 时间标定 | ✓ (核心功能) | ✓ (可选) |
| 训练时间 | ~25min (KITTI-360) | ~11min |
| 精度 (R/t) | 4.92°/64.1cm | 0.13°/8.86cm |
| 参考传感器 | LiDAR | LiDAR |
| 稳定性 | 中等 (NeRF 收敛慢) | 高 (Anchor 冻结) |

**MOISST 的核心问题**: 单一 NeRF 必须同时满足所有传感器的观测约束，当传感器视野差异大时容易产生矛盾梯度，导致收敛缓慢和精度下降。

3DGS-Calib 论文数据显示: MOISST 在 KITTI-360 上的旋转误差 4.92°±0.5° 远高于 TLC-Calib 的 0.13°±0.05°。

#### C. SOAC (Spatio-Temporal Overlap-Aware Calibration, CVPR 2024)

**核心思路**: 改进 MOISST，为每个传感器构建独立的 NeRF，并引入重叠感知损失来解决传感器间的几何一致性问题。

| 维度 | SOAC | TLC-Calib |
|------|------|-----------|
| 场景表示 | 多个 NeRF (每传感器一个) | 单一 Neural Gaussian |
| 重叠感知 | ✓ (核心创新) | 隐式 (通过 anchor 共享) |
| 训练时间 | 1-2h | ~11min |
| 参考传感器 | Camera (vs MOISST 的 LiDAR) | LiDAR |
| 开源 | ✓ | ✓ |

## 二十、GPU 内存管理架构 (rasterizer_impl)

> **源码路径**: `submodules/diff-gaussian-rasterization-w-pose/cuda_rasterizer/rasterizer_impl.h` + `.cu`

### 20.1 Chunk-Based 内存分配模式

TLC-Calib 的 CUDA 光栅化器使用**一次性大块分配 + 指针切片**的内存管理模式，避免多次 `cudaMalloc` 的开销：

```
模板函数 obtain<T>():

  static void obtain(char*& chunk, T*& ptr, size_t count, size_t alignment) {
      // 1. 对齐到 alignment (128 bytes)
      size_t offset = (chunk + alignment - 1) & ~(alignment - 1);
      // 2. 设置指针
      ptr = reinterpret_cast<T*>(offset);
      // 3. 移动 chunk 指针
      chunk = reinterpret_cast<char*>(ptr + count);
  }

使用方式:
  1. 计算总需求: size_t chunk_size = required<GeometryState>(P);
  2. 一次性分配:  char* chunkptr = geometryBuffer(chunk_size);
  3. 逐项切片:    GeometryState::fromChunk(chunkptr, P);
```

### 20.2 三大状态缓冲区详解

```
┌──────────────────────────────────────────────────────────────────────────┐
│                    GPU 内存布局 (per-frame)                               │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  GeometryState (per Gaussian, P 个)                                     │
│  ┌────────────────────────────────────────────────────────────────────┐  │
│  │ depths      : float    [P]      视图空间深度                       │  │
│  │ clamped     : bool     [P*3]    SH 颜色钳位标记                   │  │
│  │ internal_radii: int    [P]      屏幕空间半径                       │  │
```


### 21.2 投影矩阵计算

TLC-Calib 使用**基于内参的投影矩阵**而非基于 FoV 的：

```python
# graphics_utils.py
def get_projection_matrix_intrinsics(znear, zfar, cx, cy, fx, fy, W, H):
    # 从内参 (fx, fy, cx, cy) 计算 OpenGL 风格投影矩阵
    # 支持非中心主点 (cx ≠ W/2, cy ≠ H/2)

    left   = znear / fx * ((2*cx - W)/W - 1.0) * W/2
    right  = znear / fx * ((2*cx - W)/W + 1.0) * W/2
    top    = znear / fy * ((2*cy - H)/H + 1.0) * H/2
    bottom = znear / fy * ((2*cy - H)/H - 1.0) * H/2

    P = | 2n/(r-l)    0      (r+l)/(r-l)     0         |
        |    0     2n/(t-b)   (t+b)/(t-b)     0         |
        |    0        0       zf/(zf-zn)   -zf*zn/(zf-zn)|
        |    0        0          1            0          |
```

**与标准 3DGS 的差异**:
- 标准 3DGS 使用 `get_projection_matrix(fovX, fovY)` — 假设主点在图像中心
- TLC-Calib 使用 `get_projection_matrix_intrinsics(fx, fy, cx, cy)` — 支持任意主点位置
- 这对于实际传感器标定至关重要，因为真实相机的主点通常不在图像正中心

### 21.3 三个投影矩阵的关系

```python
# Camera 类的三个 property:

@property
def projection_matrix(self):
    # 纯投影矩阵 P (内参)
    return get_projection_matrix_intrinsics(...).T.cuda()

@property
def world_view_transform(self):
    # 视图变换矩阵 V (外参): world → camera
    return get_world_to_view_scaled_torch(self.cam_R, self.cam_t).T

@property
def full_proj_transform(self):
    # 组合矩阵 V × P: world → NDC
    return (V.unsqueeze(0).bmm(P.unsqueeze(0))).squeeze(0)
```

```
三个矩阵的传递关系:

  World Space ──[world_view_transform]──→ Camera Space
                     (4×4, 含 R, t)

  Camera Space ──[projection_matrix]──→ NDC Space
                     (4×4, 含 fx, fy, cx, cy)

  World Space ──[full_proj_transform]──→ NDC Space
                     (4×4, V × P)

CUDA 光栅化器接收:
  viewmatrix     = world_view_transform   (用于 frustum 剔除, Cov2D)
  projmatrix     = full_proj_transform    (用于前向投影到 NDC)
  projmatrix_raw = projection_matrix      (用于反向解析位姿梯度)
```

## 二十三、math.h 中 SE(3)/SO(3) 的 CUDA 实现解析

> **源码路径**: `submodules/diff-gaussian-rasterization-w-pose/cuda_rasterizer/math.h`

### 23.1 矩阵类型层次

```
mat33 (3×3, 列主序 float3×3)
  ├── identity()           → 单位矩阵
  ├── skew_symmetric(v)    → 反对称矩阵 [v]×
  ├── transpose()          → 转置
  ├── operator+ / * (mat) → 矩阵加法 / 乘法
  └── operator* (float3)  → 矩阵-向量乘法

mat34 (3×4, 用于 SE(3) 中间表示)
  └── mat34(mat33, float3) → [R | t]

mat44 (4×4, 齐次变换矩阵)
  ├── mat44(mat33, float3)  → 从 R, t 构造 [R t; 0 1]
  ├── mat44(mat34)          → 从 3×4 扩展
  └── operator* (mat44)    → 4×4 矩阵乘法
```

**列主序 (column-major)**: `cols[0]` 是矩阵的第一列, 与 OpenGL/CUDA 的内存布局一致。矩阵乘法 `A * B` 中, `A.cols[i]` 是 A 的第 i 列。

### 23.2 SO(3) 旋转群实现

```cpp
struct SO3 {
    mat33 data_;   // 内部存储 3×3 旋转矩阵

    // Lie algebra → Lie group: Rodrigues 公式
    static SO3 Exp(const float3 &theta) {
        mat33 W = hat(theta);        // [θ]× 反对称矩阵
        mat33 W2 = W * W;            // [θ]×²
        float angle = norm(theta);   // ||θ||
        mat33 I = mat33::identity();

        // 小角度近似 (避免 sin/cos 数值不稳定)
        if (angle < 1e-5)
            return SO3(I + W + 0.5f * W2);

        // Rodrigues 公式:
        // R = I + sin(θ)/θ · [θ]× + (1-cos(θ))/θ² · [θ]×²
        return SO3(I + sin(angle)/angle * W
                     + (1-cos(angle))/(angle*angle) * W2);
    }
};
```

**数学对应**:

```
Rodrigues 公式: R = exp([θ]×) = I + sin(||θ||)/||θ|| · [θ]×
                                + (1-cos(||θ||))/||θ||² · [θ]×²

其中 [θ]× = | 0   -θz   θy |
             | θz   0   -θx |   (反对称矩阵)
             |-θy   θx   0  |

小角度近似 (||θ|| < 1e-5):
  sin(θ)/θ ≈ 1,  (1-cos(θ))/θ² ≈ 0.5
  → R ≈ I + [θ]× + 0.5·[θ]×²
```

### 23.3 SE(3) 刚体变换群实现

```cpp
struct SE3 {
    SO3 R_data_;     // 旋转部分
    float3 t_data_;  // 平移部分

    // SE(3) 指数映射: (ρ, θ) → T ∈ SE(3)
    static SE3 Exp(const float3 &rho, const float3 &theta) {
        mat33 W = SO3::hat(theta);
        mat33 W2 = W * W;
        SO3 R = SO3::Exp(theta);
        float angle = norm(theta);
        mat33 I = mat33::identity();

        // 左雅可比矩阵 V
        mat33 V;
        if (angle < 1e-5)
            V = I + 0.5f * W + 1.f/6.f * W2;
        else
            V = I + W * ((1-cos(angle))/(angle*angle))
                  + W2 * ((angle-sin(angle))/(angle*angle*angle));

        // 平移 = V · ρ (不是直接用 ρ!)
        float3 t = V * rho;
        return SE3(t, R);
    }

    // SE(3) 组合 (左乘)
    SE3 operator*(const SE3 &T) const {
        return SE3(t + R * T.t, R * T.R);
    }

    // 逆变换
    SE3 inverse() const {
        SO3 R_inv = R.inverse();      // R^T
        float3 t = R_inv * t_data_;   // -R^T · t
        return SE3(-t, R_inv);
    }
};
```

**关键数学**: SE(3) 指数映射中, 平移部分 **不是** 直接使用 ρ, 而是通过**左雅可比矩阵 V** 变换:

```
SE(3) Exp:  T = exp([τ]∧) = | R    V·ρ |
                             | 0     1  |

左雅可比矩阵:
  V = I + (1-cos(θ))/θ² · [θ]× + (θ-sin(θ))/θ³ · [θ]×²

小角度近似:
  V ≈ I + 0.5·[θ]× + 1/6·[θ]×²

τ = [ρ; θ] ∈ se(3) 是 6 维 Lie 代数向量:
  ρ = [ρ1, ρ2, ρ3]  → 平移方向 (经 V 变换)
  θ = [θ1, θ2, θ3]  → 旋转轴角
```

### 23.4 CUDA 与 Python 双重实现对比

```
┌─────────────────────┬────────────────────────┬─────────────────────────┐
│       特性           │  CUDA (math.h)         │  Python (pose_utils.py) │
├─────────────────────┼────────────────────────┼─────────────────────────┤
│ SO3 Exp             │ SO3::Exp()             │ _SO3_exp()              │
│ SE3 Exp             │ SE3::Exp()             │ SE3_exp()               │
│ 左雅可比矩阵 V       │ SE3::Exp() 内联        │ _SO3_left_jacobian()    │
│ 反对称矩阵           │ mat33::skew_symmetric()│ _skew_symmetric()       │
│ 小角度阈值           │ 1e-5                   │ 1e-5                    │
│ 使用场景             │ 反向梯度传播            │ 前向位姿更新             │
│ 自动微分             │ ✗ (手动解析梯度)        │ ✓ (PyTorch autograd)    │
│ 精度                 │ float (32-bit)         │ float (32-bit)          │
│ 存储格式             │ 列主序 mat33/mat44     │ torch.Tensor            │
│ 矩阵乘法             │ 手写展开               │ @ 运算符                │
└─────────────────────┴────────────────────────┴─────────────────────────┘
```

**为什么需要两套实现?**
- CUDA 版本用于**反向传播**: 在 `backward.cu` 中, 需要根据 τ (Lie 代数) 构造 SE(3) 变换矩阵来计算位姿梯度 `dL/dτ`。这里的 Exp 操作是前向计算图的一部分, 梯度通过手写解析公式计算
- Python 版本用于**前向更新**: 在 `poses.py → update_delta_pose()` 中, 优化器产生 Lie 代数增量 δτ, 需要通过 `SE3_exp(δτ)` 转换为变换矩阵 ΔT, 然后左乘到当前位姿上

### 23.5 梯度传播中 SE(3) 的关键作用

```
位姿梯度传播链:

  Loss ──[dL/dC]──→ 渲染输出 C
       ──[dC/d(μ2D)]──→ 2D 投影坐标 μ2D
       ──[d(μ2D)/dτ]──→ 位姿 Lie 代数 τ ∈ se(3)

在 backward.cu 中:
  1. 构造当前位姿: SE3 T_cur(tau_data)  // 从 6 维 τ 构建
  2. 变换 3D 点:   p_cam = T_cur * p_world
  3. 计算雅可比:   d(p_cam)/dτ = [I | -[p_cam]×]  (SE(3) 伴随表示)
  4. 累加梯度:     dL/dτ += dL/d(p_cam) · d(p_cam)/dτ

其中 d(p_cam)/dτ 的解析形式:
  对于 τ = [ρ; θ]:
    d(p_cam)/dρ = R       (旋转矩阵, 平移方向梯度)
    d(p_cam)/dθ = -R·[p]×  (旋转对 3D 点的微分)
```

```python
    if self.voxel_size <= 0:
        init_dist = distCUDA2(init_points)  # CUDA KNN
        median_dist = kthvalue(init_dist, int(N*0.5))
        self.voxel_size = median_dist

    # 2. Anchor 位置 = 点云坐标 (LiDAR 坐标系)
    self._anchor = Parameter(points)

    # 3. 偏移初始化为零 (训练中学习)
    self._offset = Parameter(zeros([N, n_offsets, 3]))

    # 4. 特征初始化为零
    self._anchor_feat = Parameter(zeros([N, feat_dim]))

    # 5. 缩放 = log(sqrt(KNN距离)), 6 维 (3+3)
    dist2 = distCUDA2(points)  # 每个点到最近邻的平方距离
    scales = log(sqrt(dist2)).repeat(1, 6)  # [N, 6]

    # 6. 旋转 = 单位四元数 [1,0,0,0]
    rots = zeros([N, 4])
    rots[:, 0] = 1

    # 7. 不透明度 = inverse_sigmoid(0.1) ≈ -2.197
    opacities = inverse_sigmoid(0.1)
```

### 25.5 Anchor Growing 和 Pruning (AVC)

```
Anchor Growing 流程 (adjust_anchor + anchor_growing):

  1. 计算偏移梯度: grads = offset_gradient_accum / offset_denom
  2. 多层级生长 (update_depth=3 层):
     层 0: threshold = grad_threshold
     层 1: threshold × (hierachy_factor/2)
     层 2: threshold × (hierachy_factor/2)²

  3. 每层:
     a. 候选选择: grad >= threshold AND 观察次数足够
     b. 随机采样: 概率 = 0.5^(层+1) (越深层越少)
     c. 坐标量化: round(xyz / cur_size) → 网格坐标
     d. 去重: 排除已存在的 Anchor 网格位置
     e. 新 Anchor:
        - 位置 = 量化网格 × cur_size
        - 缩放 = log(cur_size)
        - 旋转 = [1,0,0,0]
        - 不透明度 = inverse_sigmoid(0.1)
        - 特征 = 源 Anchor 特征 (scatter_max 聚合)

Anchor Pruning:
  条件: opacity_accum < min_opacity × anchor_demon
         AND anchor_demon > check_interval × success_threshold
  即: 被频繁观察但始终低不透明度的 Anchor 被移除
```

### 26.2 光度损失 L_photo (L1)

```python
def l1_loss(network_output, gt):
    return torch.abs(network_output - gt).mean()

# 梯度分析:
# dL/dC_pred = sign(C_pred - C_gt) / (H × W × 3)
#
# 特点:
# - 对异常值鲁棒 (梯度恒定为 ±1/N)
# - 不会放大大误差 (vs L2 的线性放大)
```

### 26.3 结构相似性损失 L_ssim (D-SSIM)

```python
def ssim(img1, img2, window_size=11, size_average=True):
    # 1. 创建高斯窗口 (σ=1.5)
    window = gaussian_window(window_size=11, sigma=1.5)

    # 2. 计算局部统计量 (通过卷积)
    mu1 = conv2d(img1, window)      # 局部均值
    mu2 = conv2d(img2, window)
    sigma1_sq = conv2d(img1², window) - mu1²  # 局部方差
    sigma2_sq = conv2d(img2², window) - mu2²
    sigma12   = conv2d(img1×img2, window) - mu1×mu2  # 协方差

    # 3. SSIM 公式
    C1, C2 = 0.01², 0.03²
    ssim = (2·mu1·mu2+C1)(2·sigma12+C2) /
           (mu1²+mu2²+C1)(sigma1²+sigma2²+C2)

    # 4. D-SSIM = (1 - SSIM) / 2
    return ssim.mean()

# 梯度分析:
# dL_ssim/dC_pred 涉及:
#   - dSSIM/dmu1 (局部均值变化对 SSIM 的影响)
#   - dSSIM/dsigma12 (局部协方差对 SSIM 的影响)
#   - 通过 conv2d 反向传播, 产生空间平滑的梯度场
#
# 物理意义:
#   - L1 关注像素级误差
#   - SSIM 关注结构级误差 (亮度+对比度+结构)
#   - 组合使用: 兼顾像素精度和感知质量
#
# 权重: 0.2
```

### 26.4 尺度正则化 L_scale

```python
# train.py 中:
scaling = render_pkg["scaling"]  # [P_visible, 6]
if scaling.shape[0] > 0:
    scaling_reg = (
        clamp(scaling.max()/scaling.min() - scale_regularizer, min=0)
        .view(-1, 1).sum()
    ) / scaling.shape[0]

# 解读:
# scaling.max() / scaling.min() = 最大缩放 / 最小缩放 (条件数)
# 如果条件数 > 10 (scale_regularizer), 惩罚超出部分
# 归一化: 除以可见 Gaussian 数量
```

```
尺度正则化的梯度贡献:

  当 max/min > 10 时:
    dL_scale/d(scaling_max) = 1/min / N_visible
    dL_scale/d(scaling_min) = -max/min² / N_visible

  物理意义:
    防止 Gaussian 变得过于各向异性 (一个方向极长, 另一个极短)
    → 抑制 "needle-like" 退化
    → 保持椭球形状的合理性
    → 间接改善位姿梯度质量 (各向异性 Gaussian 的位姿梯度不稳定)

  阈值 10 的含义:
    允许最大/最小缩放比例为 10:1
    超过则线性惩罚
    权重 λ_scale=1.0 (与光度损失同量级)
```

```
┌─────────────┬──────────────────────┬───────────────────────────────────┐

├─────────────┼──────────────────────┼───────────────────────────────────┤
```

```
    - 减少迭代次数 (当前 30000, 可探索 early stopping)
    - 轻量化 Gaussian 表示 (减少 Anchor 数量)
    - TensorRT 加速 CUDA 内核
    - 使用预训练初始化 (减少收敛时间)

瓶颈 2: 需要 LiDAR 点云地图
  现状: 需要预先构建聚合地图 (SLAM/Odometry)
  改进方向:
    - 支持增量式点云输入
    - 集成轻量 SLAM 前端
    - 使用深度预测替代 LiDAR 初始化

瓶颈 3: 仅支持针孔相机
  现状: 不支持鱼眼/广角/全景相机
  改进方向:
    - 扩展 CUDA 内核支持畸变模型
    - 预去畸变 (简单但损失信息)
    - 采用通用相机模型 (如 UCM/DS)

瓶颈 4: 新车型需手动适配
  现状: 需要准备标准目录结构 + 注册先验
  改进方向:
    - 自动检测传感器配置
    - 统一数据接口 (ROS bag / Protobuf)
    - 提供 GUI 配置工具

瓶颈 5: 单次标定 (非持续监控)
  现状: 每次标定独立, 无法追踪漂移
  改进方向:
    - 集成时序外参监控
    - 外参漂移检测 + 自动触发重标定
    - 在线滤波 (EKF/UKF) 与 TLC-Calib 混合
```

## 二十八、实验结果与分析

### 28.1 KITTI-360 标定精度

| 场景 | 成功率 | 旋转误差(°) | 平移误差(cm) |
|------|--------|------------|-------------|
| Straight | 100% | 0.11±0.04 | 12.7±1.43 |
| Small zigzag | 100% | 0.15±0.02 | 10.1±0.73 |
| Large zigzag | 100% | 0.09±0.03 | 6.17±2.27 |
| Small rotation | 100% | 0.21±0.03 | 6.21±1.67 |
| Large rotation | 100% | 0.09±0.03 | 9.18±1.04 |
| **平均** | **100%** | **0.13±0.05** | **8.86±2.90** |

vs. 最强 Baseline (INF)：SR 74.5%, R 0.41°, t 32.6cm

### 28.2 NVS 性能

| 数据集 | PSNR↑ | SSIM↑ | LPIPS↓ |
|--------|-------|-------|--------|
| KITTI-360 均值 | **26.39** | **0.85** | **0.09** |
| WAYMO 均值 | **27.04** | **0.86** | **0.11** |
| FAST-LIVO2 均值 | **21.90** | 0.656 | 0.205 |

### 28.3 训练效率

| 方法 | 训练时间 | GPU 显存 |
|------|---------|---------|
| CalibAnything | >5h | - |
| INF | >4h | - |
| RobustCalib | >1h | - |
| 3DGS-Calib* | ~0.15h | RTX 5090 |
| **TLC-Calib** | **~0.18h** | **<8GB (RTX 4090)** |

### 28.4 消融实验关键发现

| 消融实验 | 结论 |
|---------|------|
| Rig vs Per-image | 去掉 rig: 0.13°→1.94°, 8.86cm→64.7cm (崩溃) |
| Auxiliary Gaussians | 去掉辅助: NVS 质量显著下降, 标定精度退化 |
| AVC | 固定 voxel: 不同场景表现不一致 |
| L_scale | 去掉正则化: 训练不稳定 |
| 全图 vs 裁剪 | 全图更优: 辅助高斯利用了 LiDAR 盲区 |
| 外参泛化 | Scene A 优化→Scene B 迁移成功 |

## 三十、关键设计洞察总结

```
┌──────────────────────────────────────────────────────────────────────────┐
│                    TLC-Calib 十大关键设计洞察                              │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  ❶ 冻结 vs 可学习的精妙分工                                               │
│     Anchor 位置冻结 → 全局尺度不漂移                                      │
│     MLP 参数可学习 → 局部几何+外观自适应                                   │
│     解决了"位姿优化导致场景漂移"和"场景过拟合导致位姿困在局部最优"的双重矛盾  │
│                                                                          │
│  ❷ 全图监督 > 区域裁剪                                                    │
│     辅助高斯扩展到 LiDAR 盲区 → 天空/建筑顶部也提供梯度                     │
│     消融实验 (Tab. VI): 不裁剪图像获得最佳标定精度                          │
│                                                                          │
│  ❸ Rig 优化的约束效应                                                     │
│     所有帧共享外参 → 每次更新影响所有帧 → 泛化性更好                        │
│     消融实验 (Tab. V): 去掉 rig 后旋转误差 0.13°→1.94° (15×)              │
│                                                                          │
│  ❹ 位姿优化的 "先稳定后释放" 策略                                          │
│     Warmup 5 轮 → 权重衰减前半段 → 余弦退火 → Refine 冻结位姿              │
│     每个阶段都有明确的设计目的,形成完整的训练策略                            │
│                                                                          │
│  ❺ CUDA 级别的解析位姿梯度                                                 │
│     四条独立路径在 GPU 上并行计算 → 比自动微分更高效                        │
│     SE(3)/SO(3) 运算在 CUDA 中实现 → 几何一致性保证                        │
│                                                                          │
│  ❻ 自适应体素控制的实用性                                                  │
│     不同场景规模 → 不同最优 anchor 密度                                     │
│     AVC 自动适配 → 无需手动调参                                            │
│                                                                          │
│  ❼ 场景-位姿解耦的增量累积                                                 │
│     每次迭代: step → SE3_exp → 累积 → 清零                                │
│     增量始终在原点附近 → 小角度近似有效 → 梯度更准确                        │
│                                                                          │
│  ❽ 外参泛化 = 真实传感器标定                                               │
│     Scene A 优化 → Scene B/C 迁移成功                                     │
│     证明了 TLC-Calib 不是在做场景过拟合                                     │
│     而是在真正估计传感器的物理参数                                          │
│                                                                          │
│  ❾ Tile-Based 渲染的 GPU 并行性                                           │
│     16×16 tile 划分 + 前缀和 + 基数排序 → O(N log N) 全局排序              │
│     共享内存协作加载 + 早停机制 → 最大化 GPU 利用率                         │
│     前向/反向对称设计 → 反向渲染复用前向缓冲区                              │
│                                                                          │
│  ❿ 最完整的评估生态                                                       │
│     train.py → render.py → metrics_pose.py → metrics_nvs.py             │
│     → full_eval.py (批量聚合) + Web Viewer (实时可视化)                    │
│     是 3DGS 标定领域评估工具链最完整的开源实现                              │
└──────────────────────────────────────────────────────────────────────────┘
```
